# 랜딩페이지 기획

이 문서는 `fabbit` 대외 사이트의 메인 랜딩 페이지를 정의한다. 목적은
브랜드 소개가 아니라, 초기 타겟 고객에게 Fabbit의 문제 해결 범위를 짧게
설명하고 파일럿/데모 리드를 확보하는 것이다.

현재 문서 기준으로 Fabbit은 "AI 추출 도구"가 아니라 **설계부터 생산 연결까지
확장하는 제조 데이터 플랫폼**으로 포지셔닝한다. 다만 대외 첫 화면에서는
`PLM`, `MES`, `온톨로지`보다 고객 언어인 `도면`, `BOM`, `변경`, `생산`을
우선 사용한다.

## 1. 문제와 목표

랜딩 페이지는 아래 4가지 문제를 동시에 해결해야 한다.

1. `PLM` 인지도가 낮아 제품 분류어만으로는 방문자가 가치를 이해하지 못한다.
2. Fabbit의 차별점은 AI 추출 하나가 아니라 **추출 후 구조화와 연결**인데,
   이 차이를 짧은 스크롤 안에서 전달해야 한다.
3. 아직 대규모 고객 로고나 공개 레퍼런스가 부족하므로, 현재 구현 화면과
   문제-해결 구조를 중심으로 신뢰를 만들어야 한다.
4. 초기 목적은 즉시 결제보다 **무료 파일럿/데모 신청** 확보와 수요 검증이다.

페이지 목표는 아래와 같이 둔다.

| 목표           | 설명                                           | 성공 기준                     |
| -------------- | ---------------------------------------------- | ----------------------------- |
| 문제 공감 형성 | 방문자가 "우리 회사 문제"라고 느끼게 한다      | 문제 섹션 도달률, 스크롤 깊이 |
| 포지셔닝 전달  | 비링커, ERP, 기존 PLM과 다른 이유를 이해시킨다 | 핵심 기능 섹션 체류 시간      |
| 타겟 선별      | 적합한 리드와 비적합 리드를 구분한다           | 폼 제출 후 적합 리드 비율     |
| 전환 확보      | 파일럿/데모 신청을 만든다                      | CTA 클릭률, 폼 제출률         |

## 2. 주요 사용자

초기 랜딩 페이지는 아래 고객군을 우선 겨냥한다.

| 사용자                   | 현재 문제                                        | 랜딩 페이지에서 확인하려는 것          | 권장 CTA           |
| ------------------------ | ------------------------------------------------ | -------------------------------------- | ------------------ |
| 대표 / 공장 책임자       | 도면·BOM이 엑셀/폴더에 흩어져 운영이 불안정      | 도입이 쉬운지, 투자 대비 효과가 있는지 | `무료 파일럿 신청` |
| 설계 / 생산 팀장         | 설계 변경이 현장에 늦게 반영되고 기준본이 흔들림 | 도면, BOM, 변경 관리가 실제로 되는지   | `제품 데모 요청`   |
| 설계 엔지니어 / 생산기술 | 수동 입력, 검색, 버전 혼선이 반복됨              | 실제 화면이 업무를 얼마나 줄여주는지   | `제품 데모 요청`   |

우선 공략 세그먼트는 아래와 같다.

- 업종: `C25(금속가공)`, `C29(기계장비)`, `C28(전기장비)`
- 규모: 종사자 `10~49명` 우선, `50~299명` 확장
- 조건: ERP는 있거나 도입 예정, PLM/MES는 없거나 약함
- 니즈: 도면/BOM은 엑셀·폴더, 설계변경과 생산 반영이 끊어짐

우선 제외 대상은 아래와 같다.

- `1~9명` 영세기업
- 이미 엔터프라이즈 PLM이 깊게 정착한 대기업
- 도면/BOM 사용 비중이 낮은 비이산형 제조업

## 3. 내비게이션과 CTA 구조

랜딩 페이지는 1페이지 스크롤 구조를 기본으로 한다. 초기 단계에서는 페이지 수를
늘리기보다 한 페이지 안에서 문제, 해결, 신뢰, 전환을 닫는 편이 낫다.

### 권장 헤더 구조

- 로고
- `문제`
- `핵심 기능`
- `도입 대상`
- `요금`
- `FAQ`
- 우측 CTA: `무료 파일럿 신청`

### 권장 1차 메시지

| 구분      | 권장 문구                                                                                                                                                               | 사용 위치                  |
| --------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------- |
| H1        | `도면·BOM 관리부터 생산 연결까지, 엑셀 없이 한 번에`                                                                                                                    | 히어로                     |
| 서브카피  | `Fabbit은 중소 제조업의 설계·생산 데이터를 AI로 연결하는 제조 데이터 플랫폼이다. 도면 업로드만으로 BOM을 구조화하고, 설계 변경과 현장 실행이 이어지는 기준본을 만든다.` | 히어로                     |
| 짧은 소개 | `설계부터 생산까지 연결하는 제조 데이터 플랫폼`                                                                                                                         | 헤더, 메타 설명, 공유 카드 |

### CTA 원칙

- 1차 CTA는 `무료 파일럿 신청`
- 2차 CTA는 `제품 데모 요청`
- 셀프서브 온보딩과 결제 흐름이 준비되면 `무료 Starter 시작`을 실험안으로 올린다.
- `지금 가입`, `지금 결제`는 초기 우선순위에서 제외
- `Starter 무료`는 보여주되, 셀프서브 결제보다 상담형 전환을 우선

> [!info] CTA 판단
>
> 현재 목적은 자격 있는 제조업 리드를 모으는 것이다. 무료 가입 수 자체보다
> 파일럿 적합 고객을 확보하는 편이 중요하므로, CTA도 `가입`보다 `신청`과
> `데모` 중심으로 둔다. 다만 온보딩과 결제 준비가 끝난 뒤에는
> `무료 Starter 시작`을 상단 CTA로 A/B 테스트할 수 있다.

### 2026-03-07 기준 참고 구성

`https://whatap.io/ko` 랜딩 페이지는 아래 리듬이 명확하다. Fabbit도 이 구조를
참고하되, 카피와 시각 언어는 그대로 복제하지 않고 제조 데이터 문맥에 맞게
번역한다.

1. 강한 히어로와 즉시 보이는 CTA
2. 히어로 직후 대표 시나리오를 한 번 더 강조하는 스포트라이트 블록
3. 플랫폼 소개 후 핵심 기능 카드 묶음
4. 적용 범위나 지원 대상을 한눈에 보여주는 배지/칩 영역
5. 신뢰 근거와 제품 증거 섹션
6. 고객 사례 또는 사례에 준하는 스토리 블록
7. 마지막 전환 CTA

## 4. 핵심 시나리오

### 시나리오 1. 차가운 유입 → 파일럿 신청

1. 방문자가 검색/커뮤니티/추천 링크로 유입한다.
2. 히어로에서 "도면·BOM 관리" 문제 해결 서비스임을 즉시 이해한다.
3. 문제 섹션과 Before/After 흐름을 보고 자사 운영 방식과 겹친다고 느낀다.
4. 현재 구현 화면과 도입 대상 섹션을 확인한 뒤 `무료 파일럿 신청`을 누른다.

### 시나리오 2. 문제 공감은 있지만 확신이 부족한 경우

1. 방문자가 기능 섹션에서 AI 추출, 구조화 BOM, 변경 추적을 확인한다.
2. `현재 구현`과 `다음 단계` 구분을 보고 과장되지 않았다고 느낀다.
3. FAQ와 요금 섹션에서 도입 방식과 ERP 관계를 확인한다.
4. `제품 데모 요청`으로 전환한다.

### 시나리오 3. 재방문 리드 → 상담 전환

1. 이전에 소개를 들은 방문자가 직접 URL로 재방문한다.
2. 바로 도입 대상, 요금, FAQ를 확인한다.
3. 적합하다고 판단하면 파일럿/데모 폼을 제출한다.

## 5. 화면 기획

### 5.1 히어로

**목적**: 제품 분류를 설명하기 전에 고객 문제와 해결 범위를 한 문장으로
이해시킨다.

- 헤드라인: `도면·BOM 관리부터 생산 연결까지, 엑셀 없이 한 번에`
- 서브카피:
  `도면 업로드만으로 BOM을 구조화하고, 설계 변경과 현장 실행이 이어지는
기준본을 만든다. Fabbit은 중소 제조업을 위한 제조 데이터 플랫폼이다.`
- 보조 배지:
  - `AI 도면 분석`
  - `구조화 BOM`
  - `변경 추적`
  - `생산 연결`
- CTA:
  - 기본: `무료 파일럿 신청`
  - 보조: `제품 데모 요청`
- 비주얼:
  - 좌측: Before/After 다이어그램 또는 핵심 카피
  - 우측: 실제 제품 화면 합성 이미지
  - 단, `작업지시`, `생산실적`이 실제 구현 전이라면 `준비 중` 배지로 표시

### 5.2 대표 시나리오 스포트라이트

**목적**: 히어로 직후에 Fabbit의 가장 강한 진입 시나리오 하나를 다시
강조한다. `WhaTap`의 추천 서비스 배너처럼, 방문자가 "무엇을 먼저 보여주는
제품인지"를 바로 이해하게 만든다.

- 권장 제목: `도면 업로드만 하면 BOM이 시스템에 쌓입니다`
- 권장 설명:
  `PDF/DWG 도면을 업로드하면 AI가 부품표를 읽고, 결과를 Excel 파일이 아니라
Fabbit 안의 구조화 BOM으로 남긴다. 추출 이후의 관리까지 한 흐름으로
이어진다.`
- 권장 구성:
  - 좌측: 짧은 문장 2~3개와 CTA
  - 우측: AI 도면 인식 화면 1장
  - 하단 미니 플로우: `도면 업로드 → AI 인식 → 구조화 BOM 등록`
- 권장 CTA: `제품 데모 요청`

### 5.3 문제 제시 섹션

**목적**: 방문자가 "우리도 이렇다"라고 느끼게 한다.

권장 제목: `문제는 도면과 BOM이 따로 노는 순간 시작됩니다`

노출할 핵심 문제는 아래 4개로 압축한다.

| 문제                | 고객 언어                                            | 근거                                       |
| ------------------- | ---------------------------------------------------- | ------------------------------------------ |
| BOM 수동 입력       | "도면 보면서 엑셀로 다시 치는 데 너무 오래 걸린다"   | 건당 30분~1시간                            |
| 도면 검색/버전 혼선 | "최신본 찾는 데 시간이 다 간다"                      | 업무시간 25~30% 검색                       |
| 설계변경 반영 지연  | "현장은 구버전으로 생산하고 있다"                    | ECR/ECO 부재, 수일~수주 지연               |
| 생산 피드백 단절    | "불량이 나도 어떤 부품/도면 문제인지 바로 못 찾는다" | PLM+MES 단절, 소기업 MES 실질 도입률 약 4% |

### 5.4 Before/After 섹션

**목적**: 기능 설명보다 흐름 차이를 빠르게 보여준다.

- Before: `도면은 폴더`, `BOM은 엑셀`, `생산실적은 수기`, `공급사 정보는 담당자 기억`
- After: `AI 자동 추출 → 구조화 BOM → 변경 추적 → 생산 연결`
- 한 줄 요약:
  `데이터를 뽑는 것에서 끝나지 않고, 같은 시스템 안에서 관리하고 연결한다`

### 5.5 해결 방식 3단계 섹션

**목적**: Fabbit의 범위를 단계적으로 이해시키고, 현재 구현과 비전을 함께
전달한다.

| 단계  | 방문자 언어                                          | 상태        | 랜딩 페이지 표현     |
| ----- | ---------------------------------------------------- | ----------- | -------------------- |
| 1단계 | `AI가 도면에서 BOM을 자동으로 만든다`                | 구현        | 실제 화면 중심       |
| 2단계 | `뽑은 BOM을 시스템 안에서 관리하고 변경을 추적한다`  | 구현/고도화 | 기능 카드 + 화면     |
| 3단계 | `설계와 생산 데이터를 연결해 리스크를 먼저 감지한다` | 방향성      | 로드맵, 준비 중 표시 |

핵심 비교 문구는 아래처럼 짧게 붙인다.

- `AI가 뽑아주기만 하는 도구가 아니라, 뽑은 뒤에도 같은 시스템에서 관리한다`
- `Excel로 내보내는 순간 끝나는 흐름이 아니라, 구조화 BOM을 기준본으로 남긴다`

권장 카드 제목:

- `AI 추출로 시작`
- `구조화 BOM으로 기준본 유지`
- `설계부터 생산까지 연결`

### 5.6 현재 제품 증거 섹션

**목적**: 아직 공개 레퍼런스가 적은 상태에서 제품 신뢰를 만든다.

반드시 실제 화면을 기준으로 구성한다.

| 화면                  | 전달 메시지                                   | 현재 상태    |
| --------------------- | --------------------------------------------- | ------------ |
| AI 도면 인식 결과     | 비정형 도면을 바로 데이터로 바꾼다            | 구현         |
| BOM 트리 / 부품 탐색  | Excel이 아니라 시스템 안에서 구조화 관리한다  | 구현         |
| 변경 요청 / 이슈 화면 | 설계 변경을 추적 가능한 흐름으로 관리한다     | 구현         |
| 생산 연결 개념 화면   | 설계 기준본이 생산으로 이어질 방향을 보여준다 | 기획/준비 중 |

> [!warning] 표현 주의
>
> 실제로 구현된 화면은 `구현 완료`, 다음 단계 화면은 `준비 중` 또는
> `로드맵`으로 분리한다. MES 기능을 이미 운영 중인 것처럼 보이게 하면 안 된다.

### 5.7 적용 데이터 / 업무 범위 띠

**목적**: `WhaTap`의 기술 커버리지 벽처럼, Fabbit이 현재 다루는 데이터와
확장 예정 범위를 빠르게 훑게 만든다. 이 섹션은 자세한 설명보다 범위 인지가
목적이므로 칩 또는 배지형 UI가 적합하다.

현재 노출 가능한 항목:

- `DWG/PDF 도면`
- `부품`
- `BOM`
- `공급사`
- `변경 요청`
- `프로젝트`
- `대시보드`

확장 예정 항목:

- `작업지시`
- `생산실적`
- `품질`

> [!info] 표현 원칙
>
> 이 영역은 `지원 기술 스택`보다 `다루는 제조 데이터와 업무 흐름`을 보여주는
> 방향이 낫다. `ERP 연동`, `도면 뷰어`, `생산 운영`은 현재 상태에 맞게
> `준비 중` 또는 후속 항목으로만 제한한다.

### 5.8 도입 대상 섹션

**목적**: 적합 고객을 스스로 식별하게 만든다.

권장 제목: `이런 팀이면 Fabbit이 맞습니다`

추천 카드:

- `ERP는 있는데 도면·BOM은 엑셀/폴더로 관리하는 팀`

- `품번·리비전 체계가 약해 재발주와 구버전 이슈가 반복되는 팀`

함께 노출할 비적합 대상:

- `1~9명 영세기업`
- `이미 대기업 PLM이 정착된 조직`

### 5.9 요금 / 도입 방식 섹션

**목적**: 가격 장벽을 낮추되, 제품이 가벼운 장난감이 아니라는 인식도 만든다.

- 핵심 메시지: `회사 단위 Freemium`
- 노출 방식:
  - `Starter 무료`
  - `Team 24.9만원 / Business 59.9만원`
  - `파일럿 중심 도입`
- 설명 문구:
  `설계자만이 아니라 생산·구매·품질까지 함께 써야 하므로, Fabbit은 인당 과금보다
회사 단위 과금을 택한다`

### 5.10 FAQ 섹션

초기 FAQ는 아래 질문을 우선 다룬다.

1. `ERP를 이미 쓰고 있어도 필요한가요?`
2. `PLM이나 MES를 몰라도 쓸 수 있나요?`
3. `어떤 파일 형식을 다루나요?`
4. `무료 파일럿은 어떻게 진행되나요?`
5. `생산 기능은 지금 어디까지 가능한가요?`

### 5.11 파일럿 검증 / 사례 섹션

**목적**: `WhaTap`의 고객 사례 섹션이 맡는 역할을 초기 Fabbit 상황에 맞게
대체한다. 공개 고객 사례가 쌓이기 전까지는 `파일럿 우선 대상`과 `검증할 지표`
를 보여주는 편이 정직하다.

권장 제목: `이런 제조팀부터 먼저 검증합니다`

권장 카드 예시:

- `금속가공`
  - BOM 수동 입력 시간
  - 최신본 혼선 감소
- `기계장비`
  - 변경 반영 리드타임
  - 협력사 커뮤니케이션 정리
- `전기장비`
  - 부품 추적성
  - 설계-생산 연결 확인

사례 확보 이후 교체 방향:

- 고객 로고
- 전후 수치
- 짧은 인용문
- 적용 범위

### 5.12 마지막 CTA 섹션

**목적**: 스크롤 끝에서 다시 한 번 전환을 닫는다.

- 제목: `엑셀과 폴더로 관리하던 도면·BOM을 한 번에 정리해보세요`
- 기본 CTA: `무료 파일럿 신청`
- 보조 CTA: `제품 데모 요청`
- 보조 문구:

## 8. 콘텐츠 기획

### 필수 자산

- AI 도면 인식 결과 스크린샷
- BOM 트리 / 부품 탐색 스크린샷
- 변경 요청 또는 이슈 화면 스크린샷
- Before/After 흐름 다이어그램
- `Starter 무료`, `회사 단위 과금` 설명 카드

### 있으면 좋은 자산

- 60~90초 제품 데모 영상
- 대표자 한 줄 신뢰 문구
  - 예: `국내 PLM 솔루션 기업에서 BOM 모듈을 개발한 창업자가 직접 설계`
- 인터뷰 기반 고객 언어 1~2개
- 추후 파일럿 확보 후 고객 로고 / 전후 수치

### 비주얼 방향

- 메인 톤은 `Safety Orange` + `Fast Blue` 조합을 기본으로 한다.
- 현장 안전색과 데이터 선 이미지를 결합해 `가볍지만 제조업다운` 인상을 만든다.
- 토끼 캐릭터는 전면 주인공보다 보조 모티프로 쓴다.
  - 예: 속도감을 표현하는 라인, 귀 모양을 닮은 포인트 아이콘
- 첫 화면은 캐릭터성보다 `도면 → 데이터 → 연결` 흐름이 먼저 보여야 한다.
- 사진보다 제품 화면, 다이어그램, 구조화 카드 UI 비중을 높인다.

### 검색/카피 키워드

- `도면 관리`
- `BOM 관리`
- `설계 변경`
- `제조업 SaaS`
- `ERP 전에 BOM 정리`

### 카피 규칙

- 첫 화면에서는 `PLM`, `MES`보다 `도면`, `BOM`, `변경`, `생산`을 앞세운다.
- `AI가 다 해준다` 같은 과장 표현은 피한다.
- `온톨로지`, `Agentic AI`는 하단 비전 섹션이나 FAQ에서만 제한적으로 사용한다.
- `도면에서 뽑아주기`보다 `뽑은 뒤에도 같은 시스템에서 관리`를 강조한다.

## 9. 정책과 예외

- 실제 구현 전 기능은 반드시 `준비 중`, `다음 단계`, `로드맵`으로 구분한다.
- 고객사 로고, 후기, 절감 수치는 확보 전까지 임의로 넣지 않는다.
- `무료 가입`을 전면 CTA로 두지 않는다. 초기 목적은 셀프서브보다 파일럿 전환이다.
- `PLM`이라는 용어는 FAQ나 비교 섹션에서만 보조적으로 쓴다.
- 가격은 공개하되, 결제 플로우가 준비되기 전에는 `결제` CTA를 두지 않는다.
- `정확도 90%`, `3초`, `즉시`, `100%` 같은 성능 수치는 측정 문서가 채워지기 전까지
  전면 카피에 쓰지 않는다.
- 외부 레퍼런스 사이트는 `섹션 리듬`만 참고하고, 문구 구조나 시각 표현을 직접
  복제하지 않는다.
