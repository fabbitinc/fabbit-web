# 랜딩페이지 디자인/개발 프롬프트

이 문서는 Fabbit 랜딩 페이지를 실제로 디자인하고 개발할 때 바로 붙여 넣을 수
있는 프롬프트를 정리한다. 한 번에 긴 프롬프트로 넘기는 방식보다, 화면 단위와
섹션 단위, 구현 단위로 쪼개서 사용하는 것을 기본 전략으로 둔다.

이 프롬프트들은 [[10_제품/제품-기획/site/1. 랜딩페이지-기획]]을 기준으로
작성했다. 레퍼런스 리듬은 `https://whatap.io/ko`의 2026년 3월 7일 기준
메인 랜딩 구조를 참고한다.

## 1. 사용 원칙

아래 원칙을 먼저 읽고 프롬프트를 사용한다.

- `WhaTap`의 구조 리듬만 참고하고, 카피/레이아웃/그래픽을 그대로 복제하지 않는다.
- Fabbit의 첫 화면은 `PLM`보다 `도면`, `BOM`, `변경`, `생산 연결`을 앞세운다.
- 실제 구현 범위와 준비 중 범위를 명확히 분리한다.
- `DWG/PDF 뷰어`, `생산실적 운영`, `ERP 연동 완료`처럼 아직 없는 기능은
  구현된 것처럼 표현하지 않는다.
- `정확도 90%`, `3초`, `즉시`, `100%` 같은 수치는 현재 전면 카피에 쓰지 않는다.

## 2. 공통 컨텍스트 프롬프트

아래 프롬프트는 다른 프롬프트 앞에 공통으로 붙이는 기본 컨텍스트다.

```text
당신은 B2B SaaS 랜딩 페이지를 설계하는 시니어 디자이너이자 프론트엔드 협업자다.

프로젝트: Fabbit
제품 정의: 중소 제조업의 설계·생산 데이터를 AI로 통합하는 제조 데이터 플랫폼
핵심 진입 문제: 도면은 폴더, BOM은 엑셀, 설계 변경은 메신저/메일로 전달되어 기준본이 흔들린다.
핵심 가치:
1. 도면 업로드만으로 BOM을 구조화한다
2. 추출 후에도 시스템 안에서 BOM과 변경을 관리한다
3. 설계부터 생산 연결까지 확장 가능한 데이터 기준본을 만든다

타깃:
- 10~49명 우선, 50~299명 확장
- 금속가공, 기계장비, 전기장비
- ERP는 있지만 도면/BOM은 엑셀·폴더로 관리하는 팀

피해야 할 것:
- PLM 용어를 첫 화면 전면에 배치
- AI 추출만을 유일한 차별점처럼 과장
- 구현되지 않은 MES 기능을 운영 중처럼 표현
- Whatap 스타일 직접 복제

원하는 결과:
- 신뢰감 있는 B2B SaaS 톤
- 제조업 맥락이 느껴지는 구조적 디자인
- 제품 화면 중심의 설득
- 무료 파일럿 신청과 제품 데모 요청으로 전환
```

## 3. 화면 단위 프롬프트

### 3.1 메인 랜딩 전체 디자인 프롬프트

이 프롬프트는 디자이너 AI나 Figma 생성형 도구에 전체 페이지 구조를 맡길 때
사용한다.

```text
아래 조건으로 Fabbit 메인 랜딩 페이지를 디자인해 주세요.

[목표]
- 첫 방문자가 5초 안에 "도면/BOM 관리와 설계-생산 연결을 돕는 제품"이라고 이해하게 만들기
- 무료 파일럿 신청 또는 제품 데모 요청으로 전환시키기

[레퍼런스 방향]
- whatap.io/ko 메인 랜딩의 정보 배치 리듬을 참고해 주세요.
- 즉, 강한 히어로 → 대표 시나리오 스포트라이트 → 핵심 기능 묶음 → 범위/지원 영역 표시 → 신뢰/제품 증거 → 사례 또는 파일럿 검증 → 마지막 CTA 흐름을 사용해 주세요.
- 단, 시각 스타일은 복제하지 말고 Fabbit만의 제조 데이터 제품처럼 재해석해 주세요.

[브랜드 방향]
- Safety Orange + Fast Blue 중심
- 배경은 단순 흰색 한 장보다, 얇은 데이터 라인이나 도면 그리드 느낌이 은은하게 보이도록 구성
- 캐릭터는 전면 사용하지 말고, 속도감/귀 모양 포인트 정도로만 활용

[필수 섹션]
1. 헤더
2. 히어로
3. 대표 시나리오 스포트라이트
4. 문제 제시
5. Before/After
6. 해결 방식 3단계
7. 현재 제품 증거
8. 적용 데이터 / 업무 범위
9. 도입 대상
10. 요금 / 도입 방식
11. FAQ
12. 파일럿 검증 / 사례 placeholder
13. 마지막 CTA

[카피 톤]
- 복잡한 제조 IT 용어 대신 고객 업무 언어 사용
- 짧고 명확한 문장
- 과장 금지

[CTA]
- 1차: 무료 파일럿 신청
- 2차: 제품 데모 요청

[제약]
- 구현 완료: AI 도면 인식, BOM 트리/부품 탐색, 변경 요청/이슈, 프로젝트, 대시보드
- 준비 중: 작업지시, 생산실적, 품질
- 미구현 기능은 반드시 준비 중으로 시각적으로 구분

[산출물]
- 데스크톱 기준 전체 랜딩 레이아웃
- 각 섹션의 역할 설명
- 추천 타이포 스케일
- 색상/간격 시스템
- CTA 위치와 반복 전략
```

### 3.2 모바일 랜딩 전체 디자인 프롬프트

```text
Fabbit 메인 랜딩 페이지를 모바일 우선으로 다시 설계해 주세요.

[목표]
- 데스크톱 구조를 축약하지 말고 모바일에서 더 빠르게 이해되도록 재배열
- 히어로, 대표 시나리오, 제품 증거, CTA가 첫 3스크린 안에 들어오도록 구성

[유의점]
- 메뉴는 햄버거로 단순화
- 히어로 비주얼은 한 장의 합성 이미지보다 핵심 화면 1장 + 짧은 배지로 정리
- 카드형 섹션은 좌우 스와이프보다 세로 스택 우선
- 가격표와 FAQ는 접고 펼치는 구조 허용
- 마지막 CTA는 sticky button 또는 하단 고정 CTA 여부까지 함께 제안

[산출물]
- 모바일 390px 기준 와이어프레임
- 각 섹션 축약 규칙
- 버튼 우선순위
- 스크롤 흐름 요약
```

### 3.3 리드 폼 / 신청 화면 프롬프트

```text
Fabbit 랜딩 페이지의 전환용 신청 폼 화면을 설계해 주세요.

[목표]
- 단순 문의 폼이 아니라 리드 선별 폼으로 설계
- 작성 피로는 낮추고, 파일럿 적합도는 판단 가능하게 만들기

[필수 입력 항목]
- 회사명
- 업종
- 직원 수
- 역할
- 현재 관리 방식 (엑셀 / 폴더 / ERP / 기타)
- ERP 사용 여부
- 가장 큰 문제
- 연락처

[구조]
- 상단: 왜 이 정보를 받는지 설명
- 중단: 2단계 또는 3단계 폼
- 우측 또는 하단: 파일럿 대상 안내 카드

[톤]
- 영업 압박처럼 보이지 않게
- "파일럿 적합 여부를 빠르게 확인하고 데모를 준비하기 위해 받는 정보"라는 맥락 전달

[산출물]
- 폼 레이아웃
- 필드 우선순위
- 에러/완료 상태
- 제출 완료 화면 문구
```

## 5. 섹션 단위 디자인 프롬프트

### 5.1 히어로 섹션 프롬프트

```text
Fabbit 랜딩 페이지의 히어로 섹션만 집중해서 디자인해 주세요.

[핵심 메시지]
도면·BOM 관리부터 생산 연결까지, 엑셀 없이 한 번에

[서브 메시지]
도면 업로드만으로 BOM을 구조화하고, 설계 변경과 현장 실행이 이어지는 기준본을 만든다.

[필수 요소]
- 헤드라인
- 서브카피
- 보조 배지 4개 내외
- 기본 CTA: 무료 파일럿 신청
- 보조 CTA: 제품 데모 요청
- 실제 제품 화면 합성 이미지

[원하는 인상]
- 무거운 제조 SI 느낌보다 빠르고 명확한 SaaS 느낌
- 그래도 지나치게 일반적인 스타트업 랜딩처럼 보이지 말 것
- 구조와 신뢰가 느껴져야 함

[금지]
- 3D 추상 구체 남발
- 뜬금없는 AI 우주 이미지
- 구현되지 않은 생산 화면을 메인 히어로 화면처럼 제시
```

### 5.2 대표 시나리오 스포트라이트 프롬프트

```text
히어로 바로 아래에 들어갈 Fabbit 대표 시나리오 스포트라이트 섹션을 디자인해 주세요.

[의도]
- 방문자가 "이 제품은 도면 업로드 후 BOM이 시스템에 쌓이는 경험을 가장 먼저 보여주는구나"라고 이해하게 만들기
- whatap.io/ko의 히어로 직후 추천 서비스 블록처럼, 하나의 강한 시나리오를 다시 강조

[핵심 문장]
도면 업로드만 하면 BOM이 시스템에 쌓입니다

[필수 구성]
- 왼쪽: 짧은 설명 2~3문장
- 오른쪽: AI 도면 인식 결과 화면
- 하단 또는 내부: 도면 업로드 → AI 인식 → 구조화 BOM 등록 미니 플로우
- CTA: 제품 데모 요청
```

### 5.3 문제 제시 섹션 프롬프트

```text
Fabbit 랜딩 페이지의 문제 제시 섹션을 설계해 주세요.

[핵심 문제]
1. BOM 수동 입력
2. 도면 검색/버전 혼선
3. 설계변경 반영 지연
4. 생산 피드백 단절

[원하는 표현]
- 단순 텍스트 목록보다 문제 카드 4개 또는 2x2 그리드
- 각 카드에는 고객 언어 한 줄 + 짧은 설명
- 공감이 먼저, 기술 설명은 나중

[톤]
- 위기 마케팅 과장 금지
- "이 문제를 아는 사람만 바로 반응하는" 제조 실무 톤
```

### 5.4 Before/After 섹션 프롬프트

```text
Fabbit 랜딩의 Before/After 섹션을 설계해 주세요.

[Before]
- 도면은 폴더
- BOM은 엑셀
- 생산실적은 수기
- 공급사 정보는 담당자 기억

[After]
- AI 자동 추출
- 구조화 BOM
- 변경 추적
- 생산 연결

[원하는 구성]
- 좌우 비교형 또는 세로 전환형
- 데이터가 흩어진 상태에서 하나의 기준본으로 모이는 느낌
- 화살표와 카드 흐름이 명확해야 함
```

### 5.5 해결 방식 3단계 프롬프트

```text
Fabbit의 해결 방식 3단계 섹션을 디자인해 주세요.

[단계]
1. AI 추출로 시작
2. 구조화 BOM으로 기준본 유지
3. 설계부터 생산까지 연결

[표현 원칙]
- 1, 2단계는 비교적 선명하게
- 3단계는 비전/확장성 영역으로 보이게
- 구현 완료와 방향성의 시각 차이를 만들 것

[넣을 비교 문구]
- AI가 뽑아주기만 하는 도구가 아니라, 뽑은 뒤에도 같은 시스템에서 관리한다
- Excel로 내보내는 순간 끝나는 흐름이 아니라, 구조화 BOM을 기준본으로 남긴다
```

### 5.6 현재 제품 증거 섹션 프롬프트

```text
Fabbit 랜딩 페이지의 제품 증거 섹션을 설계해 주세요.

[목적]
- 공개 고객 사례가 적은 상태에서 제품 그 자체로 신뢰를 만들기

[노출 화면]
- AI 도면 인식 결과
- BOM 트리 / 부품 탐색
- 변경 요청 / 이슈 화면
- 준비 중: 생산 연결 개념 카드

[원하는 결과]
- 단순 스크린샷 나열이 아니라, 각 화면이 어떤 문제를 해결하는지 캡션으로 연결
- 준비 중 항목은 "로드맵" 또는 "다음 단계" 배지 표시
```

### 5.7 적용 데이터 / 업무 범위 섹션 프롬프트

```text
Fabbit 랜딩 페이지의 적용 데이터 / 업무 범위 섹션을 설계해 주세요.

[의도]
- whatap.io/ko의 서비스 커버리지 묶음처럼, Fabbit이 현재 다루는 범위를 한눈에 보여주기
```
