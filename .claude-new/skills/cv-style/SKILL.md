# Style Conventions

## 목표

- 스타일링과 횡단 관심사를 일관된 규칙으로 유지하라.
- 토큰 기반 디자인 시스템을 강제하라.
- 이 스킬은 스타일/횡단 관심사 규칙만 다루고 다른 레이어의 코드 작성 규칙은 다루지 마라.

## 토큰 체계

### 패키지 경계

| 위치 | 역할 |
|---|---|
| `packages/theme` | CSS 변수 정의 (토큰 원천) |
| `packages/ui` | CSS 변수 소비 (`var(--*)`) |
| `apps/web` | CSS 변수 소비 + 테마 적용 |

### 색상 토큰 규칙

- 컴포넌트에 hex/rgb를 직접 사용하지 마라. 반드시 CSS 변수(`var(--*)`)로 참조하라.
- 새 색상 추가 시 "정의 먼저, 사용 나중" 원칙을 따르라.
- 추가 순서: `packages/theme` -> `apps/web` CSS 파일 -> 컴포넌트에서 사용.
- 의미 기반 네이밍을 사용하라 (`--brand-*`, `--status-*`, `--theme-*`). 용도 불명 이름을 금지한다.
- 특정 테마에만 필요한 토큰을 단독 추가하지 마라. 모든 테마에 동일 키로 제공하라.

### 테마 적용 규칙

- 기본 fallback: `:root`와 `.dark`에 모두 추가하라 (삭제하지 않고 최소 안전값 유지).
- 브랜드 계열: `theme-primary-*` 모든 변형에 추가하라.
- 공통 상태(success/warning/danger/info): `theme-common-*` 모든 변형에 추가하라.
- 실제 운영 값은 `theme-primary-*`, `theme-common-*`에서 우선 관리하고, `:root`/`.dark`는 fallback 역할로 유지하라.
- 테마 작업 후 `/dev/design`에서 primary/common 전환으로 시각 검증하라.

## Tailwind 사용 규칙

- `cn()` 유틸리티로 className을 병합하라.
- 인라인 스타일은 CSS 변수 참조가 필요한 최소 범위에서만 사용하라.
- 컴포넌트는 가능하면 클래스 기반으로 스타일을 적용하라.
- 반응형: mobile-first(`sm:`, `md:`, `lg:`) 순서로 작성하라.
- 다크 모드: `dark:` 접두사를 직접 사용하지 마라. CSS 변수가 테마별로 전환되므로 변수만 참조하면 된다.

## 에러 처리 (횡단 관심사)

### 에러 바운더리

- 페이지 단위로 에러 바운더리를 배치하라.
- 에러 바운더리는 fallback UI와 재시도 버튼을 제공하라.
- 에러 바운더리 내부에서 API 호출이나 복잡한 로직을 수행하지 마라.

### API 에러 처리 계층

| 계층 | 처리 대상 |
|---|---|
| HTTP 클라이언트 인터셉터 | 401(인증 만료 → 로그아웃), 403(권한 부족), 500(서버 오류) |
| Action hook | 도메인 특화 에러 (409 충돌, 422 유효성 등) |
| 에러 바운더리 | 미처리 에러 (예상 외 런타임 에러) |

- API 레이어에서 에러를 삼키지 마라. 상위로 전파하라.
- 에러 메시지는 사용자 친화적으로 변환하라. 서버 에러 메시지를 그대로 노출하지 마라.

## 사용자 피드백 (횡단 관심사)

### 토스트

- 성공/실패 피드백은 토스트로 표시하라.
- 토스트 트리거는 action hook에서 수행하라. 컴포넌트에서 직접 트리거하지 마라.
- 토스트 메시지는 행동 결과를 명확히 전달하라 ("이슈가 생성되었습니다", "삭제에 실패했습니다").

### 로딩 상태

- 로딩 상태는 query hook/action hook의 `isLoading`/`isPending`을 사용하라.
- 전역 로딩 스피너를 남용하지 마라. 영역별 스켈레톤/스피너를 우선하라.
- 뮤테이션 중 중복 요청을 방지하라 (`isPending` 상태에서 버튼 비활성화).

### 확인 다이얼로그

- 파괴적 작업(삭제, 아카이브)은 확인 다이얼로그를 먼저 표시하라.
- 확인 다이얼로그는 packages/ui의 ConfirmDialog 프리미티브를 사용하라.

## UX 일관성 규칙

- 클릭 가능한 모든 요소에 `cursor-pointer`를 적용하라.
- 비활성 요소에 `opacity-50`과 `pointer-events-none`을 적용하라.
- 폼 제출 중 버튼에 로딩 상태를 표시하라.
- 빈 상태(empty state)에 안내 메시지와 행동 유도(CTA)를 제공하라.

## 적용 절차

1. 새 색상/토큰이 필요하면 `packages/theme`에 먼저 정의하라.
2. `:root`/`.dark` fallback과 `theme-*` 변형에 모두 추가하라.
3. 컴포넌트에서 CSS 변수로 참조하라.
4. 에러 처리 계층(인터셉터/action hook/에러 바운더리)을 확인하라.
5. 사용자 피드백(토스트/로딩/확인)이 적절히 배치됐는지 점검하라.
6. `/dev/design`에서 테마 전환 시각 검증하라.

- 에러 바운더리가 페이지 단위로 배치됐는가?
- API 에러를 삼키지 않고 상위로 전파하는가?
- 토스트가 action hook에서 트리거되는가?
