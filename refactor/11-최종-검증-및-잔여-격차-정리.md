# 11 최종 검증 및 잔여 격차 정리

## 작업 목표

- [x] 이번 세션에서 완료된 `web -> packages -> web2` 이관 범위를 사실대로 정리한다.
- [x] 빌드/스토리북 검증 결과를 기록한다.
- [x] 아직 남은 실질 격차를 다음 작업 단위로 분리한다.

## 완료 범위

- `apps/web`의 앱 셸(`MainLayout`, `Header`, `Sidebar`) 기준 구조를 `packages/components`의 `AppShell`, `AppHeader`, `AppSidebar`로 다시 정리했다.
- `apps/web2/src/app/layouts/main-app-layout.tsx`는 패키지 셸을 조립하고, 라우팅/인증/조직 전환/로그아웃 연결만 남기도록 바꿨다.
- `apps/web`의 생성 화면(`CreateForm`, `CreateFormSidebar`)을 기준으로 `packages/components/src/change-create-screen.tsx`를 추가했다.
- `apps/web2/src/features/change-shared/components/change-create-form.tsx`는 lookup query와 submit 연결만 담당하도록 축소했다.
- `apps/web2/src/features/issue/components/issue-detail-screen.tsx`의 `useAuthStore` selector 렌더 루프 위험을 제거했고, 상세/타임라인 재시도를 함께 수행하도록 수정했다.
- `apps/storybook`의 `AppShell`, `AppHeader`, `AppSidebar`, 페이지 스토리를 새 public import 기준으로 맞췄고 `ChangeCreateScreen` 스토리를 추가했다.

## 검증 결과

- `pnpm lint:web2` 통과
- `pnpm build:web2` 통과
- `pnpm build-storybook` 통과

## 잔여 격차

- `issue detail`, `change request detail`은 현재 `packages/components`를 사용하지만, `apps/web`의 `ChangeRequestDetailPage` 계열 마크업을 기준으로 완전히 다시 승격한 상태라고 보기 어렵다. 시각 구조와 상태 조합을 더 엄격하게 `web` 기준으로 덮어써야 한다.
- `settings`, `onboarding`, `auth`는 package 소비 구조가 이미 넓게 적용돼 있지만, 실제 화면 기준의 마지막 패리티 확인은 별도 브라우저 비교가 더 필요하다.
- 실제 사용자 화면 비교 기준의 최종 완료 판단은 아직 이 문서에서 내리지 않는다. 이번 세션은 앱 셸, 생성 화면, 렌더 루프 안정화, 패키지 public import 정리까지를 완료 범위로 본다.
